Wireshark - Network protocol analyzer

Redesigning an OSX application

Role: UX/UI Designer

Wireshark is a great, free open-source packet analyzer used for network troubleshooting, analysis, software and communications protocol development, and education. I first came across Wireshark in college while taking a networking course. In the beginning, while I was still learning the ins and outs of the tool, I found myself stuck performing the same tasks each time. I thought there had to be more to it; its intended use, I thought, had to be more than just scanning and randomly following TCP streams. That’s when I started reading its documentation and found how the GUI came to be and why Wireshark was not simply another command line packet sniffer like tcpdump. I understood the main focus wasn’t its user interface, but the powerful functionality packed into it. However if the tool was meant for education, I thought some things needed to be changed to enable to user with an easier, more accessible tools.

Ready to Scan

One of the first things I felt needed to be modified was Wireshark’s dependency on its documentation to perform simple tasks. A three-column design, for instance, created a somewhat cluttered/confusing interface that obscured the starting objective – performing a scan.

I opted for a dropdown instead that automatically selects the most active interface as default, and a call-to-action button that proceeds with the scan. Nonetheless, a savvier user could perform more detailed scans by simply selecting the advanced scan option.

Visual Organization (Less is More)

Organizing the interface to follow a more logical structure seemed to be the top priority. After all, having 4 sections in one screen was confusing and impractical as one could not concentrate on either one of them. I focused on the packet capture and got rid of 2 sections. I also incorporated a left panel with some useful tools -- extracted from the top -- used to find and filter packets.

Intelligent Search and Filtering

I believe search is a way to discover the capabilities of an app. Creating an experience around it is, then, just as important as redesigning the entire front-end composition.

The search field was used as way to filter packets through prebuilt expressions (i.e. “atn-cm contains 12” or simply “dns”). I opted for a friendlier approach to search, which could be accessed from either the left panel or from the search box. If for example I wanted to filter out all the ‘HTTP’ and ‘DNS’ packets, I’d simply go on the left panel click on the “add sign” and proceed to adding protocol filters which could later be easily removed.

Packet Isolation

As mentioned before, the old interface was initially composed of 4 sections; the two bottom ones expressed a selected packet’s content. When I removed these two sections I visualized a more targeted packet analysis, which for now comes in the form of tabs.
